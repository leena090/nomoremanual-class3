## ① HTML 디자인 파일 만들기

### 왜 HTML을 먼저 만들까?

바로 앱을 만들면 디자인이 뒤죽박죽됩니다.
**먼저 디자인을 HTML 한 파일로 완성**하고, 그걸 "레퍼런스"로 사용하는 게 핵심입니다.

### 클로드에게 이렇게 말하세요

```
내 강의 판매용 랜딩페이지를 HTML 한 파일로 만들어줘.

[내 강의 정보]
- 강의명: ○○○ 마스터클래스
- 타겟: (누구를 위한 강의인지)
- 가격: ○○○원 (정가 ○○○원에서 할인)
- 선착순 ○○명 마감
- 커리큘럼: (간략히)
- 후기: (있다면)

[디자인 요청]
- 타겟이 50~60대이므로 버튼 크고, 글씨 크게
- 깔끔하고 신뢰감 있는 디자인
- 섹션: 히어로 → 고민 공감 → 해결책 → 강사소개 → 커리큘럼 → 후기 → 가격 → 환불보장 → FAQ → CTA → 푸터
```

### 결과

`landing.html` 파일이 생성됩니다.
브라우저에서 열어보고, **마음에 들 때까지 수정을 요청**하세요.

```
"히어로 문구를 바꿔줘: ○○○로"
"가격 섹션에 할부 안내 추가해줘"
"후기를 2개 더 추가해줘"
```

**이 HTML이 완벽하다고 느낄 때까지 수정하세요.**
이 파일이 이후 모든 개발의 기준이 됩니다.

## ② PRD (기획서) 작성

### PRD가 뭐야?

"이 사이트가 뭘 할 수 있어야 하는지" 정리한 문서입니다.
클로드에게 구체적인 지시를 주기 위한 설계도입니다.

### 클로드에게 이렇게 말하세요

```
이 landing.html을 실제 결제가 되는 풀스택 웹앱으로 만들려고 해.
PRD(제품 요구사항 문서)를 작성해줘.

필요한 기능:
1. Next.js로 포팅 (HTML 디자인 100% 유지)
2. 토스페이먼츠 결제 연동
3. Supabase 데이터베이스 (수강생 정보 저장)
4. 관리자 페이지 (신청자 목록, CSV 다운로드)
5. 결제 완료 시 이메일 자동 발송
6. 잔여석 실시간 표시 (하드코딩 금지)
7. 신청 폼: 이름, 이메일, 전화번호, 관심트랙, 사용경험, 수강목표

환경변수로 API 키 관리 (하드코딩 금지)
API 키가 없을 때는 Mock 모드로 동작
```

### PRD에 포함되는 내용 (클로드가 자동 작성)

| 섹션 | 내용 |
|------|------|
| 기술 스택 | Next.js 14, TypeScript, Tailwind CSS, Supabase, 토스페이먼츠, Resend |
| 디자인 시스템 | 색상 코드, 폰트, 사이즈 기준 |
| 페이지 구조 | `/` (랜딩), `/success`, `/fail`, `/admin` |
| DB 스키마 | enrollments 테이블 컬럼 정의 (SQL 포함) |
| API 설계 | 6개 API Route와 각각의 역할 |
| 결제 플로우 | 고객 → 폼 → 결제 → 승인 → DB → 이메일 |
| 환경변수 목록 | 필요한 API 키 11개 |
| 이메일 템플릿 | 결제 완료 시 발송할 이메일 내용 |
| 신청 폼 필드 | 6개 필드의 타입, 필수 여부, 검증 규칙 |

> **이 PRD 파일을 저장해두세요** — 다음 단계에서 사용합니다.

  - CSV 다운로드
  - 검증: 필터/CSV 동작 확인

Phase 4: 이메일 + 배포
  - Resend 이메일 발송
  - Vercel 배포
  - 환경변수 설정
  - 검증: 프로덕션 URL에서 E2E 테스트

> **핵심**: 디자인 → PRD → 계획 → 승인 → 구현. 이 순서가 품질을 보장합니다.

## ⑥ Supabase DB 연결

### 클로드에게 이렇게 말하세요

```
Supabase 세팅해줘
```

### 클로드가 자동으로 하는 일

1. **조직 확인** → 기존 조직 사용
2. **비용 확인** → 무료 ($0/월) → 대표님께 확인
3. **프로젝트 생성** → `nomoremanual-landing` (서울 리전)
4. **테이블 생성** → SQL 자동 실행 (enrollments 테이블)
5. **API 키 자동 설정** → URL + anon key를 `.env.local`에 반영

### 대표님이 직접 할 일 (1가지)

**service_role 키를 Supabase 대시보드에서 복사**

1. Supabase 대시보드 → ⚙️ Settings → API
2. `service_role` (secret) → Reveal → 복사
3. 클로드에게 붙여넣기

```
sb_secret_XXXXXXXX...
```

클로드가 `.env.local`에 자동 반영합니다.

> **이 키는 왜 필요해?**
> service_role 키가 있어야 서버에서 DB를 읽고 수정할 수 있습니다.
> anon 키로는 데이터 넣기만 가능하고, 읽기/수정은 안 됩니다.

## ⑧ Resend 이메일 연동

결제 완료 시 수강생에게 **확인 이메일을 자동 발송**하는 서비스입니다.

### 설정 방법

1. https://resend.com 가입 (GitHub 계정으로 가능)
2. API Keys → Create API Key → 복사
3. 클로드에게 알려주기:

```
re_XXXXXXXX...
```

클로드가 `.env.local`에 자동 반영합니다.

> Resend가 없어도 결제/신청은 정상 동작합니다. 이메일만 안 나갈 뿐.

## ⑩ 결제 테스트

### 클로드에게 이렇게 말하세요

```
결제 테스트 해보자
```

클로드가 dev 서버를 재시작합니다 (환경변수 반영).

### 테스트 순서

1. http://localhost:3000 접속
2. **"2기 신청하기"** 버튼 클릭
3. 폼 입력:
   - 이름: 테스트
   - 이메일: test@test.com
   - 전화: 010-1234-5678
   - 트랙: 아무거나

## 핵심 교훈 5가지

### 1. "디자인 → PRD → 계획 → 승인 → 구현" 순서를 지켜라
이 순서를 지키면 결과물 품질이 완전히 달라집니다.
바로 "만들어줘"하면 엉망이 됩니다.

### 2. HTML 디자인 파일이 모든 것의 기준
landing.html이 "정답지" 역할을 합니다.
클로드가 이걸 보고 동일한 디자인을 코드로 만듭니다.

### 3. Mock 모드로 먼저 전체 완성, 서비스 연결은 나중에
API 키 없이도 전체 UI가 동작합니다.
UI가 완성된 후에 Supabase/토스/Resend를 연결하는 게 효율적입니다.

### 4. 키는 대화로 전달하면 끝
"이 키 넣어줘" → 클로드가 `.env.local`에 설정 → 끝.
설정 파일을 직접 건드릴 필요 없습니다.

### 5. git push = 자동 배포
Vercel + GitHub이 연결되면, 수정 후 push만 하면 자동으로 배포됩니다.
"배포"라는 개념을 몰라도 됩니다.
